# 核心概念

理解這些核心概念將幫助您在平台上建立有效的自動化工作流程。

## 階層概覽

平台使用階層結構來組織您的自動化資源：

```
代理
  └── 工作區
       ├── 模型
       │    ├── 表單（欄位 + 記錄）
       │    ├── 表單
       │    └── 動作
       └── API 連接
```

## 代理

### 什麼是代理？

**代理**是平台中最高層級的組織單位。將它視為一個完整的自動化專案或業務功能。

### 特徵
- 包含多個工作區
- 代表廣泛的自動化領域
- 將資源與其他代理隔離
- 可以獨立管理和共享

### 範例
- 行銷自動化代理
- 客戶支援代理
- 內容創作代理
- 銷售管道代理
- HR 入職代理

### 何時建立多個代理
建立獨立的代理當：
- ✅ 處理完全不同的業務功能
- ✅ 需要嚴格的資料分離（不同客戶）
- ✅ 需要不同的權限結構
- ✅ 建立不同的產品或服務

使用新工作區而不是新代理當：
- ❌ 只是組織不同的活動
- ❌ 分離不同的內容類型
- ❌ 測試新功能
- ❌ 建立臨時專案

## 工作區

### 什麼是工作區？

**工作區**是代理內的頂層組織容器。它將相關模型分組並管理 API 連接。

### 核心屬性
- 屬於恰好一個代理
- 包含多個模型
- 有自己的 API 連接集
- 為不同專案或環境提供隔離

### 全域工作區概念

每個工作區都作為其模型的「全域範圍」：
- 工作區內的模型可以相互參照
- 動作可以存取工作區中任何模型的欄位
- API 連接在所有模型之間共享
- 表單可以在任何工作區模型中建立記錄

### 範例
- 生產環境
- 測試環境
- 活動 A
- 活動 B
- Q1 專案
- 客戶專案

### 工作區限制
- **每種 API 類型一個連接** - 您可以有一個 Facebook 連接、一個 Instagram 連接等，但不能有同一類型的多個連接
- 模型範圍限定在工作區
- 排程在工作區邊界內運作

## 模型

### 什麼是模型？

**模型**定義資料結構 - 您想要追蹤的資訊以及它是如何組織的。

### 模型的元件
1. **架構定義** - 欄位類型和屬性
2. **表單** - 實際的資料表格（記錄和值）
3. **表單** - 用於建立記錄的輸入介面
4. **動作** - 資料的處理工作流程

### 模型範例

#### 電商產品模型
```
欄位：
- 產品名稱（文字）
- 描述（長文字）
- 價格（數字）
- 庫存數量（數字）
- 類別（選擇）
- 圖片（檔案上傳）
- 已發布（核取方塊）
```

#### 客戶模型
```
欄位：
- 姓名（文字）
- 電子郵件（電子郵件）
- 電話（電話）
- 公司（文字）
- 狀態（選擇：潛在客戶、活躍、流失）
- 最後聯絡（日期）
- 備註（長文字）
```

### 模型之間的關聯

模型可以相互參照：
- **一對多**：一個客戶有多個訂單
- **多對多**：產品可以有多個標籤，標籤適用於多個產品
- **父子**：動作可以在相關模型中建立子記錄

## 表單

### 什麼是表單？

**表單**是儲存您實際資料的資料庫表格。每個模型恰好有一個表單。

### 表單結構
- **欄位**（列）- 定義要捕獲的資料
- **記錄**（行）- 表格中的個別項目
- **檢視** - 顯示和篩選資料的不同方式

### 欄位類型

| 類型 | 描述 | 範例 |
|------|-------------|---------|
| 文字 | 短文字字串 | 產品名稱 |
| 長文字 | 多行文字 | 描述、備註 |
| 數字 | 數值 | 價格、數量 |
| 核取方塊 | 布林真/假 | 已發布、活躍 |
| 選擇 | 預定義選項 | 狀態、類別 |
| 日期 | 日期值 | 建立日期 |
| 日期/時間 | 帶時間的日期 | 最後更新 |
| 電子郵件 | 電子郵件地址 | 聯絡電子郵件 |
| 電話 | 電話號碼 | 客戶電話 |
| URL | 網址 | 網站連結 |
| 檔案 | 檔案上傳 | 圖片、文件 |
| 關聯 | 連結到其他模型 | 客戶 ID |

### 處理記錄

每筆記錄代表一個單一實體：
- 透過表單或 API 建立
- 透過動作處理
- 手動或自動更新
- 透過關聯連結到其他記錄

## 欄位

### 什麼是欄位？

**欄位**是表單中的列 - 它們定義每筆記錄包含的資料。

### 欄位屬性
- **名稱**：人類可讀的識別符
- **類型**：資料類型（文字、數字、日期等）
- **必填**：欄位是否必須有值
- **預設值**：新記錄的預填值
- **驗證規則**：可接受值的限制

### 特殊欄位用途

#### 輸入欄位
儲存使用者輸入或從 API 接收的原始資料的欄位：
```
- 標題
- 描述
- 使用者電子郵件
- 原始內容
```

#### 輸出欄位
儲存 AI 處理或 API 回應結果的欄位：
```
- 生成的說明
- 圖像 URL
- API 回應
- 處理狀態
```

### 全域欄位

在動作的上下文中，「全域工作區欄位」是指工作區中任何模型的欄位都可以用作輸入或輸出。

**範例**：
```
工作區：行銷中心
  模型 A：部落格貼文
    - 標題
    - 內容
  
  模型 B：社交貼文
    - 平台
    - 貼文文字
    
模型 B 中的動作可以使用模型 A 的標題和內容作為輸入
```

## 記錄

### 什麼是記錄？

**記錄**是表單中的單一行 - 您模型的一個實例。

### 記錄生命週期

1. **建立**
   - 透過表單提交
   - 透過 API
   - 透過動作（建立子記錄）
   - 手動輸入

2. **處理**
   - 動作在個別記錄上執行
   - AI 分析和生成內容
   - API 呼叫傳送/接收資料
   - 欄位被更新

3. **關聯**
   - 父子連結
   - 跨模型參照
   - 階層結構

4. **封存/刪除**
   - 軟刪除（標記為非活躍）
   - 硬刪除（移除）
   - 匯出備份

## 資料流原則

### 輸入 → 處理 → 輸出

所有自動化都遵循這個模式：

```
輸入欄位 → 動作步驟 → 輸出欄位
```

**範例**：
```
{產品名稱, 描述} → AI 增強 → {增強描述, 關鍵字}
```

### 可重用性

儲存在欄位中的資料可以：
- 在不同動作中多次使用
- 被其他模型參照
- 排程定期處理
- 匯出或與外部系統整合

### 全域 vs. 本地範圍

**本地範圍**（模型層級）：
- 模型內的欄位
- 模型表單中的記錄
- 特定於模型的表單
- 為模型定義的動作

**全域範圍**（工作區層級）：
- 工作區中的所有模型
- 所有工作區 API 連接
- 跨模型參照
- 共享資料存取

## 關聯和資料連接

### 父子關聯

動作可以建立子記錄：

```
部落格文章（父）
  └── 社交貼文 1（子）
  └── 社交貼文 2（子）
  └── 社交貼文 3（子）
```

### 跨模型參照

模型可以相互參照：

```
訂單模型：
  - 客戶（→ 客戶模型）
  - 產品（→ 產品模型，多個）
  - 運送（→ 運送模型）
```

## 核心原則

### 1. 關注點分離
- **模型**定義結構
- **表單**處理輸入
- **動作**處理資料
- **表單**儲存資料
- **排程**自動執行

### 2. 可組合性
- 結合簡單元件來建立複雜的工作流程
- 在不同動作中重用模型
- 透過關聯在模型之間共享資料

### 3. 靈活性
- 欄位可以服務多個目的
- 動作可以連結在一起
- 資料可以多向流動

### 4. 可擴展性
- 隨著需求增長新增模型
- 為組織建立額外的工作區
- 使用多個代理水平擴展
